Fabric · AI 工厂「知识库」（集成「知识本」）

把成熟的知识本能力接进 Fabric，让 Agent 拥有可精准检索的专属知识库
📅 2026-07-02 🏷️ v1.0（一期） 👤 作者：张振 🧩 归属：Fabric / AI 工厂
1背景与目标

在 Fabric 给 Agent 配置知识时，当前后台没有独立的知识库配置能力，只能把整理后的知识塞进以下 3 个地方，各有明显局限：

① 写进 Prompt

受上下文长度约束，无法承载大量知识；改一处要动 Prompt，难维护、易污染指令。

② 放云空间文档

Agent 读不到，得靠 Skill 绕路去读；格式杂、无结构化检索，命中率低。

③ 塞 Echo 知识库

需自己整理成规定格式，整理成本高；来源单一，多格式知识进不来。

缺少标准化的知识配置能力，由此带来 3 个突出问题：

  • 无标准操作、效果不一：Fabric 当前没有配置知识的能力，只能各自想办法配置，不同方法差异较大，缺乏标准操作，导致 Agent 最终呈现效果参差不齐。
  • 配置门槛高、费力度大：自己配置知识时，还要额外配置各类 Skill 去链接知识，对配置同学的能力要求高、投入也大。
  • 结构化差、检索不准：手工整理的知识结构化程度低，知识之间缺少关联关系，导致检索不准，最终影响回复效果。

而内部已有一款成熟的知识类产品 「知识本」，却无法在 Fabric 里直接使用：

  • 多源多格式上传：POPO 云空间 / 本地文件（PDF、DOCX、XLS、CSV、Markdown、PNG、MP4、SRT…）/ 网络链接 / KM 文章
  • 自动生成 Wiki：上传后自动抽取「原文摘要 / 人物与事物 / 术语解释 / 待解问题 / 方法与流程」，且 Wiki 之间带关联关系，利于关联检索
  • 价值：一站式覆盖多来源知识、大幅降低整理成本、并通过结构化 + 关联增强检索准确性

本期目标：把「知识本」作为知识底座接入 Fabric，在「AI 工厂」下新增「知识」模块，实现三件事——① 给 Agent 配专属知识库；② C 端咨询时 Agent 能精准检索知识；③ 在 POPO 租户下管理 C 端用户上传的知识与 Wiki。

2产品定位

2.1 谁来配、谁受益

角色能做什么
租户配置 / 管理人员（具备该租户权限）在「AI 工厂 · 知识」下新建知识本、上传资料、把知识本绑定到 Agent；并对本租户下 C 端用户创建的知识本 / Wiki 做统一管理

4一期范围

✅ 一期要做

  • 「AI 工厂 · 知识」菜单，具备该租户权限的成员可见
  • Fabric 内新建知识本，复用知识本原生的多源上传 + 自动 Wiki 能力
  • 知识本与 Agent 多对多绑定：一个 Agent 可选多个知识本，一个知识本也可被多个 Agent 选用；检索范围严格限定在所选集合内
  • C 端对话时，Agent 调用知识本检索能力获取知识作答
  • POPO 租户下对 C 端用户上传知识 / Wiki 的完整管理（查看 / 编辑 / 删除 / 下架 / 归属移转）
  • 复用 Skills 的分类 / 展示标签 / 可见性能力：管理分类、管理标签、按分类 / 展示标签过滤，以及每个知识本的可见性配置（逻辑同 Skills）

❌ 一期不做

  • 知识内容的编辑 / 结构维护 / 长期治理
  • 引用已有 C 端知识本作为 Agent 知识（一期只在 Fabric 内新建）
  • 关注 / 管理知识本的产物（思维导图、可视化报告、云文档、KM 文章）
  • 检索的向量库自建、检索算法调优（沉在知识本）
  • 知识本内容的版本管理与审批流
  • C 端用户与配置者之间的知识协作 / 评论
5核心流程一：给 Agent 配知识

5.1 关联模型：一个 Agent 可选多个知识本

一个租户下有 N 个 Agent，为避免"可查全部知识、检索找乱"，知识与 Agent 必须显式关联。配置 Agent 时可从本租户已建知识本中勾选一个或多个知识本；检索时只在该 Agent 所选的知识本集合内进行，未选中的知识本不参与检索：

运维 Agent
客服 Agent
— 可多选绑定 →
运维知识本被运维 Agent 选中
通用 FAQ 知识本被运维 + 客服 Agent 选中
客服知识本被客服 Agent 选中
💡 多选带来的检索范围管控
允许多选是为覆盖"专属本 + 公共本"等组合场景（如各 Agent 都需通用 FAQ）。但选得越多、跨领域知识越易互相干扰而拉低命中率，因此：① 检索范围严格限定在所选集合内，绝不扩展到全部；② 建议配置者按需精选，避免堆叠无关知识本；③ 多本检索由知识本一次调用完成合并（统一大索引 + notebook_id filter，返回全局 Top-K），Fabric 侧不做应用层重排（见 6.2）。

1

5

5.3 知识本配置字段

字段必填说明
知识本名称是便于识别，1~40 字
描述否说明该知识本覆盖的领域 / 用途

处理状态—系统态：解析中 / 已生成 Wiki / 处理失败（供绑定前判断是否就绪）
绑定 Agent—展示当前被哪些 Agent 选用（可被多个 Agent 选中）

5.4 Agent 侧「知识库」配置项

字段必填说明
绑定知识本否从本租户已建知识本中多选（0~多个）；不选则该 Agent 无知识库、走原有作答逻辑
知识优先级否检索命中时是否优先采信知识本内容（默认：优先，未命中再走模型通用能力）

🔑 合并检索原则：Fabric 只调 1 次，由知识本一次完成合并
无论 Agent 绑定 1 本还是 N 本知识本，Fabric 侧始终只调 1 次检索接口，把 notebook_ids 列表整体传给知识本；跨本召回 / 去重 / 重排 / 全局 Top-K 排序，全部在知识本内部一次完成（统一大索引 + notebook_id metadata filter）。Fabric 不做应用层"分本查询 + RRF 融合"，避免跨本分数不可比造成的精度损失。

6.1 检索时机

一期采用命中即用策略：Agent 绑定了知识本时，每轮用户提问都先做一次知识本检索；检索为空或相关度低于阈值时，回落到模型通用能力作答，并可提示"该问题未在知识库中找到明确依据"。

6.2 关键约束

  • 范围隔离：检索请求必须带上 Agent 所选的 notebook_ids，服务端二次校验绑定关系，杜绝检索到未选中 / 越权的知识本。
  • 合并检索一次完成：跨多个知识本的召回 / 去重 / 重排全部由知识本侧一次调用完成，返回全局 Top-K；Fabric 只调 1 次接口、不做应用层分本查询与重排，跨本相似度分数在同一大索引下天然可比。
  • 无绑定即降级：Agent 未选任何知识本时不发起检索，走原有作答逻辑，不报错。
⚠️ 待知识本团队确认
① 底层向量库是否为「统一大索引 + notebook_id metadata」形态，检索接口能否接受 notebook_ids 列表参数、一次返回全局 Top-K；若暂不支持，能否在接口层内部封装完成合并（Fabric 侧仍只调 1 次）；② 检索延迟与并发上限（影响 C 端体验）。
7C 端用户知识本的统一治理

POPO 是 Fabric 平台的一个租户。C 端用户在 POPO 租户下自建的知识本，与配置者创建的知识本一样，统一汇入第 3 节的知识本管理后台——并非单独的菜单 / Tab，而是复用同一套统一管理功能，只是把 C 端用户创建的知识本一并展示出来，支持增删改查。治理只针对知识与 Wiki，不关注产物（思维导图、可视化报告、云文档、KM 文章）。

💡 与 Skills 治理保持相同逻辑
POPO 租户下同样支持 C 端用户自建 Skills，创建后也在统一管理后台展示，供租户管理人员管理。知识本治理对齐这套现成逻辑：租户管理人员可对 C 端用户创建的知识本执行删除 / 更新，删除 / 更新后不通知 C 端用户（与 Skills 一致）。

8权限与依赖
  • 菜单可见性：「AI 工厂 · 知识」对具备该租户权限的成员可见 / 可操作；C 端用户在 Fabric 后台不可见。

  • 对 Agent 的依赖：知识本被某 Agent 选用时，该 Agent 删除 → 解除对应绑定关系；知识本被删除 → 从所有选用它的 Agent 知识库中移除并提示。
  • 数据隔离：检索、治理均以租户 + notebook_ids 为边界，严禁越界访问。
9交付与验收
  • 「知识」菜单出现在 AI 工厂下，具备该租户权限的成员可见，为统一的知识本管理后台。
  • 可在 Fabric 内新建知识本、多源上传、自动生成 Wiki。
  • 可给一个 Agent 勾选一个或多个知识本，且检索严格限定在所选知识本集合内。
  • C 端提问端到端跑通：Agent 调用知识本检索 → 跨所选多本召回重排 → 命中知识 → 作答。
  • 未选任何知识本的 Agent 正常降级作答，不报错。
  • C 端用户在 POPO 租户下创建的知识本，自动汇入统一后台，可增删改查；删除 / 更新不通知 C 端用户（与 Skills 一致）。
  • 产物（思维导图 / 可视化报告 / 云文档 / KM 文章）确认不在治理范围。
