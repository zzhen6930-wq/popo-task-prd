Fabric · AI 工厂「知识管理」（集成「知识本」）

把 NotebookLM 式的知识本能力接进 Fabric，让 Agent 拥有可维护、可精准检索的专属知识库
📅 2026-07-02 🏷️ v1.0（一期） 👤 作者：张振 🧩 归属：Fabric / AI 工厂
1背景与目标

在 Fabric 给 Agent 配置知识时，当前后台没有独立的知识管理模块，只能把整理后的知识塞进以下 3 个地方，各有明显局限：

① 写进 Prompt

受上下文长度约束，无法承载大量知识；改一处要动 Prompt，难维护、易污染指令。

② 放云空间文档

Agent 读不到，得靠 Skill 绕路去读；格式杂、无结构化检索，命中率低。

③ 塞 Echo 知识库

需自己整理成规定格式，整理成本高；来源单一，多格式知识进不来。

而内部已有一款成熟的知识类产品 「知识本」（高度类似 Google NotebookLM），却无法在 Fabric 里直接使用：

  • 多源多格式上传：POPO 云空间 / 本地文件（PDF、DOCX、XLS、CSV、Markdown、PNG、MP4、SRT…）/ 网络链接 / KM 文章
  • 自动生成 Wiki：上传后自动抽取「原文摘要 / 人物与事物 / 术语解释 / 待解问题 / 方法与流程」，且 Wiki 之间带关联关系，利于关联检索
  • 价值：一站式覆盖多来源知识、大幅降低整理成本、并通过结构化 + 关联增强检索准确性

本期目标：把「知识本」作为知识底座接入 Fabric，在「AI 工厂」下新增「知识」模块，实现三件事——① 给 Agent 配专属知识库；② C 端咨询时 Agent 能精准检索知识；③ 在 POPO 租户下治理 C 端用户上传的知识与 Wiki。

2产品定位

2.1 谁来配、谁受益

角色能做什么
组织配置者（管理员）在「AI 工厂 · 知识」下新建知识本、上传资料、把知识本绑定到 Agent；在 POPO 租户下治理 C 端用户上传的知识与 Wiki
C 端使用者与投放的 Agent 对话时，间接享受知识本检索带来的更准回答；不感知知识本配置细节

2.2 为什么直接复用「知识本」，而不自建知识模块

维度复用知识本（本期）Fabric 自建知识库
多源多格式上传✅ 已具备❌ 需从零建设
结构化 / Wiki / 关联检索✅ 自动生成❌ 需自研抽取与向量能力
整理成本低（自动整理）高（需人工规整）
C 端知识本已有存量✅ 可直接治理复用❌ 数据割裂
研发投入集成对接为主重复造轮子
🔑 核心策略：Fabric 不重造知识能力，只做「集成 + 关联 + 治理」
Fabric 负责三层薄封装：关联层（知识本 ↔ Agent）、检索接入层（Agent 对话时调用知识本检索）、治理层（POPO 租户下管理 C 端知识 / Wiki）。知识的存储、整理、Wiki 生成、检索算法全部沉在知识本。

4一期范围

✅ 一期要做

  • 「AI 工厂 · 知识」菜单，仅组织配置者可见
  • Fabric 内新建知识本，复用知识本原生的多源上传 + 自动 Wiki 能力
  • 一个 Agent 可选绑多个知识本，检索范围严格限定在所选知识本集合内
  • C 端对话时，Agent 调用知识本检索能力获取知识作答
  • POPO 租户下对 C 端用户上传知识 / Wiki 的完整管理（查看 / 编辑 / 删除 / 下架 / 归属移转）

❌ 一期不做

  • 多个 Agent 共享同一知识本（二期，需配套共享 / 权限设计）
  • 引用已有 C 端知识本作为 Agent 知识（一期只在 Fabric 内新建）
  • 关注 / 管理知识本的产物（思维导图、可视化报告、云文档、KM 文章）
  • 检索的向量库自建、检索算法调优（沉在知识本）
  • 知识本内容的版本管理与审批流
  • C 端用户与配置者之间的知识协作 / 评论
5核心流程一：给 Agent 配知识

5.1 关联模型：一个 Agent 可选多个知识本

一个租户下有 N 个 Agent，为避免"可查全部知识、检索找乱"，知识与 Agent 必须显式关联。配置 Agent 时可从本租户已建知识本中勾选一个或多个知识本；检索时只在该 Agent 所选的知识本集合内进行，未选中的知识本不参与检索：

运维 Agent
客服 Agent
— 可多选绑定 →
运维知识本被运维 Agent 选中
通用 FAQ 知识本被运维 + 客服 Agent 选中
客服知识本被客服 Agent 选中
💡 多选带来的检索范围管控
允许多选是为覆盖"专属本 + 公共本"等组合场景（如各 Agent 都需通用 FAQ）。但选得越多、跨领域知识越易互相干扰而拉低命中率，因此：① 检索范围严格限定在所选集合内，绝不扩展到全部；② 建议配置者按需精选，避免堆叠无关知识本；③ 多本检索结果需统一重排 / 去重后再喂给模型（见 6.2）。

5.2 配置流程

1
配置者 @ Fabric
进入「AI 工厂 · 知识」→ 新建知识本（填名称、描述、所属业务）。
2
知识本
上传资料：POPO 云空间 / 本地文件 / 网络链接 / KM 文章。知识本自动解析、生成关联 Wiki（摘要 / 人物与事物 / 术语 / 待解问题 / 方法流程）。
3
配置者 @ Fabric
进入目标 Agent 编辑页 → 「知识库」配置项 → 勾选一个或多个已建好的知识本。
4
Fabric
建立 Agent ↔ 知识本 多对关联记录（一个 agent_id 对应一组 notebook_id）。绑定校验：所选知识本须属于本租户且处理就绪；重复勾选自动去重。
5
Agent
配置完成，Agent 具备由所选知识本组成的知识库；投放到 C 端后即可基于该集合作答（见第 6 节检索流程）。

5.3 知识本配置字段

字段必填说明
知识本名称是便于识别，1~40 字
描述否说明该知识本覆盖的领域 / 用途
资料来源是至少 1 条：云空间 / 本地文件 / 网络链接 / KM 文章；沿用知识本原生上传能力与格式支持
处理状态—系统态：解析中 / 已生成 Wiki / 处理失败（供绑定前判断是否就绪）
绑定 Agent—展示当前被哪些 Agent 选用（可被多个 Agent 选中）

5.4 Agent 侧「知识库」配置项

字段必填说明
绑定知识本否从本租户已建知识本中多选（0~多个）；不选则该 Agent 无知识库、走原有作答逻辑
知识优先级否检索命中时是否优先采信知识本内容（默认：优先，未命中再走模型通用能力）
6核心流程二：C 端咨询时的检索方案

Agent 投放到 C 端后，用户提问时，采用调用知识本检索能力的方案（Fabric 不自建向量库，检索算法沉在知识本）。完整链路：

1
C 端用户
向投放的 Agent 提问，如"POPO 机器人发消息接口有字数限制吗？"
2
Agent Runtime
根据 Agent 绑定关系拿到所选知识本 ID 列表（notebook_ids），携带用户 query 调用知识本检索接口。检索范围被 notebook_ids 严格限定，只在这些知识本内检索。
3
知识本
在指定的知识本集合内做语义检索（结合 Wiki 结构与关联关系），跨多本结果统一召回、重排 / 去重后返回 Top-K 相关片段。
4
Agent
将检索片段作为上下文与用户 query 一并喂给模型，生成答案。按「知识优先级」策略决定采信程度。
5
返回用户
输出答案给用户。

6.1 检索时机

一期采用命中即用策略：Agent 绑定了知识本时，每轮用户提问都先做一次知识本检索；检索为空或相关度低于阈值时，回落到模型通用能力作答，并可提示"该问题未在知识库中找到明确依据"。

6.2 关键约束

  • 范围隔离：检索请求必须带上 Agent 所选的 notebook_ids，服务端二次校验绑定关系，杜绝检索到未选中 / 越权的知识本。
  • 多本重排：跨多个知识本召回的片段需统一相关度重排、去重后再取 Top-K，避免单本淹没或重复。
  • 无绑定即降级：Agent 未选任何知识本时不发起检索，走原有作答逻辑，不报错。
⚠️ 待知识本团队确认
① 跨多个知识本检索时，召回片段的相关度重排 / 去重由知识本侧完成，还是需 Fabric 侧处理；② 检索延迟与并发上限（影响 C 端体验）。
7核心流程三：C 端知识治理（POPO 租户）

知识本当前已作为 C 端产品对用户开放。需在 Fabric-POPO 租户下，对 C 端用户上传的知识与生成的 Wiki 进行完整管理。不关注产物（思维导图、可视化报告、云文档、KM 文章）。

💡 明确不纳入治理的范围
本期不管理知识本产物：思维导图、可视化报告、云文档、KM 文章。治理只对准"输入侧的知识（资料）"与"结构化产出的 Wiki"。
8权限与依赖
  • 菜单可见性：「AI 工厂 · 知识」仅组织配置者可见 / 可操作；C 端用户完全不可见。
  • 对知识本的依赖：整个方案依赖知识本提供 —— 新建 / 上传 / Wiki 生成能力、检索能力、C 端知识 / Wiki 的管理 API（查看 / 编辑 / 下架 / 删除 / 移转）。
  • 对 Agent 的依赖：知识本被某 Agent 选用时，该 Agent 删除 → 解除对应绑定关系；知识本被删除 → 从所有选用它的 Agent 知识库中移除并提示。
  • 数据隔离：检索、治理均以租户 + notebook_ids 为边界，严禁越界访问。
9交付与验收
  • 「知识」菜单出现在 AI 工厂下，仅管理员可见，含「知识本管理」「C 端知识治理」两 Tab。
  • 可在 Fabric 内新建知识本、多源上传、自动生成 Wiki。
  • 可给一个 Agent 勾选一个或多个知识本，且检索严格限定在所选知识本集合内。
  • C 端提问端到端跑通：Agent 调用知识本检索 → 跨所选多本召回重排 → 命中知识 → 作答。
  • 未选任何知识本的 Agent 正常降级作答，不报错。
  • POPO 租户下可对 C 端用户上传知识 / Wiki 完成查看 / 编辑 / 下架 / 删除 / 移转，操作有审计。
  • 产物（思维导图 / 可视化报告 / 云文档 / KM 文章）确认不在治理范围。
10二期方向（不在一期范围）
  • 多个 Agent 共享同一知识本（配套共享 / 权限 / 检索范围可控设计）
  • 引用 C 端已有知识本直接作为 Agent 知识（免重复新建）
  • 检索策略增强：多路召回、重排、按 Wiki 关联做扩展检索
  • 知识本内容版本管理与变更审批
  • 知识本使用分析：命中率、未命中问题沉淀（反哺补充知识）
  • 知识本产物（思维导图 / 报告等）在 Fabric 侧的呈现与复用
