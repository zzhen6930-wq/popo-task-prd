Fabric · AI 工厂「记忆」能力（一期）

为 Agent 补齐分层记忆 + 异步做梦能力，让智能体像数字同事一样越用越懂你
📅 2026-07-22 🏷️ v1.0 👤 作者：青远（张振） 🧩 归属：Fabric / AI 工厂
1背景与目标

Fabric 是面向组织 / 用户的 Agent 配置平台。当前 Agent 没有记忆能力——每次对话都是"失忆"状态，导致：

  • 用户反复交代同样的背景、偏好、身份，Agent 记不住，体验断裂
  • 用户纠正过的错误、明确过的禁忌，下次对话又犯，无法沉淀"教训"
  • 跨会话的项目上下文、决策、事实无法积累，Agent 无法越用越聪明
  • 只能靠把信息硬塞进 Prompt / Skill，规模一大就超上下文、命中率低

本期目标：为 Fabric 补齐一套分层记忆架构（基础文件 + 用户记忆）+ 异步做梦（Auto-Dream）机制，并在「AI 工厂」下新增「记忆」菜单支持后台可视化管理。核心机制对标 Claude Tag（分层文件记忆 + 异步做梦 + 按需注入），短期记忆参考 OpenClaw 的会话缓冲设计。

🔑 一句话定位
让 Agent 从「群聊摘要机器人」升级为「具备睡眠整理能力的数字同事」：对话时保持实时流畅，空闲时在"后台做梦"消化信息，沉淀为可复用的长期知识。
2核心概念与术语
术语说明
记忆（Memory）Agent 持久化存储的信息，以文件为载体（Markdown 结构化文本）
基础文件相对静态的底座文件，含 soul（智能体人格设定）与 user（用户身份档案）两个文件
用户记忆随交互动态积累的记忆，含短期记忆与长期记忆

长期记忆跨会话持久化的知识，由多个分类文件组成

3记忆分层架构（总览）

记忆分为 两大分类：基础文件 与 用户记忆。用户记忆再分短期与长期，长期记忆由多个分类文件组成。

📁 基础文件（相对静态）
soul 人格
智能体的名字、性格、沟通风格。定义"我是谁、我怎么说话"。以人工配置为主，做梦可克制补充。
user 身份+偏好
用户档案：基础信息 / 偏好设置 / 工作背景 / 知识技能 / 重要关系 / 近期动态。定义"你是谁、你喜欢怎样"。做梦持续克制丰富。
🧠 用户记忆（动态积累）
短期记忆 · 每日笔记 Daily Notes
原始对话细致提炼后的每日记录（保留大量细节、按时间线），落盘 memory/YYYY-MM-DD.md。按需加载进上下文，也是做梦的原材料。
长期记忆 多分类文件
由多个分类文件组成（见第 5 节）：事实记忆 / 反馈记忆 / 引用记忆。
💡 基础文件 vs 长期记忆：分工
基础文件（soul / user）= 关于"人"本身的稳定画像（智能体人格 + 用户身份与偏好），做梦克制回填，克制增长；长期记忆= 关于"事"的可复用知识（项目事实、踩坑禁忌、外部引用），做梦持续沉淀。用户的偏好习惯归入 user 文件的「偏好设置」，长期记忆不再单列偏好类型。
4基础文件详解

4.1 soul 文件（智能体人格）

用户场景：配置者希望 Agent 有稳定的名字、性格与说话风格（如"叫小法，严谨、引用条款、少废话"），不因对话漂移。

解决问题：人格散落在 Prompt 里易被覆盖、难维护；沉淀为独立文件后稳定、可复用、可版本管理。

内容：主要记录三项——智能体名字、性格、沟通风格。

机制：soul 在每次对话必注入系统上下文最前部；以人工配置为主，做梦仅在识别到明确的人格/风格调整时做克制补充（如用户长期反馈"你太啰嗦" → 收敛沟通风格），不频繁改写、不发散。

4.2 user 文件（用户身份 + 偏好档案）

用户场景：用户不想每次重复自己是谁、喜欢怎样的回答、在做什么项目。

解决问题：身份与偏好类信息高频复用，硬塞 Prompt 浪费上下文；集中沉淀为一份结构化档案，随交互越来越懂用户。

内容结构（Markdown 分段，参考示例）：

分段示例内容
基础信息称呼 / 时区 / 语言 / 城市
偏好设置沟通风格（简洁直接）/ 回复长度 / 工作时间 / 免打扰时段
工作背景职位 / 当前项目 / 关注领域
知识技能擅长 / 学习中 / 常用工具
重要关系关键协作人及关系（如技术负责人、直属上级）
近期动态最近的关键事件（带日期）

机制：user 在对话时必注入；支持用户 / 管理员人工编辑，做梦克制增量丰富（补充新识别的偏好、动态、关系），与人工内容冲突时不静默覆盖（见 6.3）。

⚠️ 克制原则
soul 与 user 会随聊天越来越丰富，但必须克制：只沉淀高置信、稳定、可复用的画像；一次性、情绪化、临时性内容不写入基础文件，避免档案膨胀失焦。
5用户记忆详解

5.1 短期记忆 · 每日笔记（Daily Notes）

用户场景：

  • 对话记录：「我想查看昨天和智能体的完整对话」
  • 工作回顾：「这周都讨论了哪些内容？帮我汇总」
  • 知识沉淀：「把今天的讨论整理到长期记忆」
  • 问题追溯：「上周遇到的那个问题是怎么解决的？」

解决问题：原始对话零散且量大，直接堆进上下文既溢出又低效；自动整理成每日笔记，既能回顾检索，又作为长期记忆的原材料。

机制：短期记忆是平台的异步整理机制，把每日对话细致提炼后按时间线保存到 memory/YYYY-MM-DD.md。整理提炼不在对话中实时进行，而是在会话上下文压缩前自动触发 memory flush——提醒 Agent 将本段重要上下文落盘到当日每日笔记，再继续压缩。

🔑 关键：短期记忆是"提炼产物"，但提炼 ≠ 做梦
短期记忆不是原始对话流水，而是原始对话经一道细致提炼后的每日笔记——保留了大量细节、按时间线组织。这道提炼异步触发：当会话上下文将被压缩时，先触发 memory flush，把重要上下文落盘为每日笔记，只做结构化整理、不压缩细节；而做梦是每晚运行的另一道提炼，把每日笔记压缩去重、抽取规则后写入长期记忆。二者时机、强度、产物都不同：每日笔记（详）→ 长期记忆（精）。

整理提炼的时机

  • 会话压缩前自动触发 memory flush：当前会话上下文接近容量、需要压缩时，系统先自动触发一次 flush
  • 提醒 Agent 落盘：flush 时提醒 Agent 将本段重要上下文写入当日 memory/YYYY-MM-DD.md，避免压缩丢失关键信息

每日笔记记录内容

  • 日期和时间戳
  • 用户输入和智能体回复
  • 重要的工具调用和结果
  • 用户明确要求"记住"的事项

5.2 长期记忆（多分类文件）

用户场景：Agent 要记住"项目用 PostgreSQL 15，升级前须跑 X 脚本""上次因为漏了权限校验踩过坑""详细规范在某文档"。

解决问题：把碎片信息沉淀为分类清晰、可复用的知识，而非一坨扁平摘要。

机制：长期记忆由多个分类文件组成，各文件更新节奏不同：

长期记忆文件内容来源作用更新频率
事实记忆
Fact / Project		项目决策、技术约定、里程碑、稳定事实记住"我们在做什么、怎么做才对"每晚做梦整合
反馈记忆
Feedback		用户的否定、修正、边界与禁忌记住"什么不能做、上次错在哪"强制即时记录（不等做梦）
引用记忆
Reference		外部文档链接、API 规范、知识库索引知道"去哪里找详细信息"按需添加
💡 引用而非背诵
对长文档 / 大段讨论，长期记忆只存索引 / 链接到「引用记忆」，需要时再读取原文（可对接知识本 RAG），而非把原文总结硬塞进核心记忆——控制记忆体积、提升命中率。
⚠️ 反馈记忆为什么"强制即时"
禁忌 / 纠正类信息若等到做梦才记，可能在下一轮就重复犯错。因此反馈记忆在识别到的当下即时写入，不进"待做梦"队列。
6异步做梦机制（Auto-Dream · 核心）

用户场景：用户白天和 Agent 聊了很多（已整理为每日笔记），每晚 Agent 自动"消化"当天笔记，第二天变得更懂用户。

解决问题：每日笔记保留了大量细节、会持续堆积；每晚做梦时对当日笔记做一道压缩提炼，得到高价值长期知识，控制体积、沉淀规则。

6.1 触发时机

  • 每晚定时触发（主）：每日固定时段（类比人类睡眠 REM）批量处理当日每日笔记——这是一期唯一的常规触发方式，不做空闲/实时提炼
  • 手动触发（辅）：配置者可在后台「记忆」菜单手动触发一次做梦，用于调试或即时整理

短期记忆
  • 压缩与去重：碎片对话压缩成高密度结构化笔记（Markdown），剔除冗余，抽取可复用的规则 / 决策 / 事实
  • 分类归档：提炼结果按语义分流——用户偏好/身份/关系/动态克制回填 user，人格/风格调整克制回填 soul，项目事实入事实记忆，禁忌入反馈记忆，外部资料入引用记忆
  • 合并策略：同类信息增量合并而非覆盖；与已有条目冲突时保留更新的、标注时间戳；基础文件回填遵循克制原则，只写高置信稳定内容
  • 做梦日志：每次做梦产出一条 Dream Log（处理了哪些、沉淀了什么、写入了哪些文件），后台可查、可追溯

6.3 与人工编辑的防冲突

  • 用户 / 管理员在后台人工编辑过的记忆条目优先级更高，做梦不静默覆盖人工内容
  • 做梦对人工条目只做补充 / 建议，冲突项标记待确认，避免"越改越乱"
🔑 知识内化闭环
感知（对话）→ 记录（会话压缩前异步 flush，整理成每日笔记，保留细节）→ 按需加载（检索相关笔记进上下文）→ 触发（每晚定时做梦）→ 做梦提炼（压缩去重、抽取规则）→ 固化（回填 soul/user + 写长期记忆）→ 应用（下次直接调用，无需重读全部历史）。反馈/禁忌为例外，识别当下即时固化，不等做梦。
7记忆的读取与注入（按需注入）

对话发生时，按以下优先级组装上下文，只注入相关片段，不全量背诵：

记忆注入策略
soul必注入，置于系统提示最前
user必注入（身份 + 偏好，直接决定回答风格）
短期记忆（每日笔记）按需加载：意图分析（关键词 / 时间范围 / 话题）→ 相关性检索（语义 / 关键词 / 时间过滤）→ Top-K 筛选去重截断 → 作为「历史笔记」片段并入上下文（详见 5.1）
长期记忆（事实 / 反馈）按当前问题相关性检索注入 Top-K 片段；反馈记忆（禁忌）优先注入
引用记忆默认只注入索引；命中需要时再拉取原文（可走知识本 RAG）
💡 按需加载 ≠ 全量背诵
不是每次都把全部历史塞进上下文，而是根据当前对话内容智能检索、只加载相关的每日笔记与长期记忆片段，既省上下文窗口又提高相关性。soul / user 因体量小且高频复用，采用必注入。

9典型场景串联
场景 1 · 偏好沉淀
用户多次要求"回答简洁点"
当天对话整理进每日笔记（短期记忆）→ 每晚做梦提炼出"用户偏好简洁、要点式回答" → 回填 user 文件的「偏好设置」 → 次日对话必注入 user，无需再交代。
场景 2 · 禁忌即时记
用户说"以后别用外部 API 示例，我们内网不通"
识别为反馈 / 禁忌 → 即时写入反馈记忆（不等做梦）→ 下一轮起优先注入，避免重复犯错。
场景 3 · 引用而非背诵
用户丢来一篇 50 页技术规范
长期记忆只在引用记忆存"规范链接 + 摘要索引" → 用户问到相关问题时按需拉取原文（对接知识本 RAG），不把全文塞进记忆。
场景 4 · 后台治理
管理员发现一条记忆记错了
进入「记忆」菜单 → 选中 Agent → 用户记忆 Tab → 编辑 / 删除对应长期记忆条目；该人工修改优先于后续做梦，不被静默覆盖。
10一期范围

✅ 一期要做

  • 分层记忆架构：基础文件（soul / user，含用户偏好）+ 用户记忆（短期=每日笔记 / 长期）
  • 短期记忆=每日笔记（细致提炼、按时间线、memory/YYYY-MM-DD.md）+ 按需加载
  • 长期记忆三类分类文件：事实 / 反馈 / 引用（偏好并入 user）
  • 异步做梦：每晚定时对每日笔记做梦提炼归档（回填 soul·user + 写长期记忆）+ Dream Log
  • 反馈记忆强制即时记录
  • 按需加载注入（意图分析 + 相关性检索 Top-K + 引用只存索引）
  • 基础文件做梦克制回填，不静默覆盖人工内容
  • 「记忆」菜单：Agent 列表 + 详情横向 Tab（基础文件 / 用户记忆）
  • 记忆文件的查看 / 编辑 / 删除 + 手动触发做梦
  • 人工编辑 vs 做梦的防冲突策略

❌ 一期不做

  • 记忆的版本回溯 / 时间线 diff（二期）
  • 跨 Agent / 跨用户的记忆共享与继承
  • 记忆条目级的细粒度权限与审批流
  • 做梦策略的可视化编排（自定义提炼规则）
  • 记忆容量计费与配额后台
  • 多模态记忆（图片 / 音频记忆）
11权限与依赖
  • 菜单可见性：「记忆」菜单对具备该空间权限的配置者可见；记忆数据按 Agent + 用户维度隔离
  • 对模型的依赖：做梦的提炼、注入的相关性检索依赖 LLM + 向量检索能力
  • 对调度的依赖：每晚定时做梦依赖调度能力（可复用「自动化任务」调度引擎）
  • 对知识本的依赖：引用记忆的原文拉取可对接知识本 RAG（第 7 节）
  • 数据安全：记忆含用户敏感信息，须加密存储、访问审计、支持一键清除
